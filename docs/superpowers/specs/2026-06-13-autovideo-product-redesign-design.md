# AutoVideo 产品版重做设计

日期：2026-06-13

## 背景

AutoVideo 是从 `/Users/sha/junxincode` 中拆出的独立视频混剪产品。新仓库不做旧系统的简单搬迁，而是重做产品边界和模块结构：复用源项目里已经验证过的视频渲染、字幕、BGM、音色处理经验，但不继承旧 WebUI 的登录态、个人网盘导入、内网配置和真实凭据。

当前仓库已使用 `AGPL-3.0-only`。第一版目标是让个人用户能在本地或自托管环境中完成视频混剪任务，并能管理混剪所需的字幕模板、BGM 和音色资源。

## 目标

1. 建立独立产品架构，目录清晰，便于后续接入权限管理。
2. 提供可用的混剪工作台：选择素材、配置脚本/字幕/配音/BGM、创建任务、查看状态、下载输出。
3. 提供资源中心能力：字幕模板库、BGM 管理、音色中心。
4. 所有配置通过环境变量和示例配置表达，不把真实 token、key、密码、内网地址写入仓库。
5. 前端用户可见文案使用中文，并同时设计桌面端和移动端的可用布局。
6. 保持视频导出结果以 FFmpeg、ASS/libass、现有混剪算法为主要真值来源，避免重新发明核心渲染链路。

## 非目标

第一版不做以下能力：

1. 登录、权限管理、团队空间、租户隔离。
2. 个人网盘、NAS 登录态导入、旧 WebUI 的用户态下载链接。
3. 素材 AI 索引、大规模向量检索、自动场景理解作为必需能力。
4. 版权 BGM 商店、在线素材采购、支付和商业授权流程。
5. 原样搬迁旧页面、旧路由和旧配置文件。

这些能力可以在独立边界稳定后作为后续阶段接入。

## 产品结构

前端采用 React + Vite + TypeScript 编写，构建后作为静态 SPA 部署。首版提供以下一级入口：

1. 混剪工作台：核心任务创建与进度查看。
2. 素材库：本地素材目录、上传素材、在线素材候选结果。
3. 字幕模板：模板列表、模板编辑、预览、导入导出。
4. BGM 管理：上传、分类、试听、重命名、删除。
5. 音色中心：音色列表、试听、Edge TTS / Fish Speech 配置状态、声音复刻入口。
6. 任务与输出：历史任务、失败原因、输出下载。
7. 系统设置：外部服务地址、目录状态、运行依赖检查。

桌面端使用左侧导航 + 主工作区 + 右侧预览/状态栏。移动端改为顶部主入口 + 横向二级标签 + 底部关键操作区，避免依赖 hover 和过密按钮。

## 后端架构

首版后端采用 FastAPI，按职责拆成以下模块：

1. `api`：路由层，只处理请求校验、响应模型、错误映射。
2. `core`：配置、路径、日志、错误类型、运行依赖检查。
3. `services/video_mix`：混剪任务编排、素材准备、渲染调用、任务状态。
4. `services/subtitle`：字幕模板、ASS 生成、模板预览、DSL 配置转换。
5. `services/bgm`：BGM 文件管理、分类、试听元数据、删除和重命名。
6. `services/voice`：Edge TTS、Fish Speech 音色列表、试听、复刻任务适配。
7. `storage`：文件目录、元数据 JSON 或轻量数据库、输出文件管理。
8. `workers`：后台任务执行入口，第一版可以先用进程内任务队列，后续再替换为 Redis/RQ/Celery。
9. `web`：托管 React/Vite 构建产物，并在本地开发时支持前端 dev server 代理。
10. `tests`：单元测试、接口测试、前端组件测试和静态构建检查。

路由层不能直接堆业务逻辑。源项目中 `video.py` 这类大路由文件只作为功能参考，不作为新仓库的结构模板。

## 数据与文件布局

默认运行数据放在可配置的数据目录下，例如：

```text
data/
  materials/
  bgm/
  voices/
  subtitle_templates/
  outputs/
  tasks/
```

每类资源有独立元数据文件或轻量数据库表。第一版优先使用文件系统 + JSON 元数据，原因是部署成本低、易排查、适合个人使用。需要并发写入的资源操作必须使用文件锁或原子写入，避免上传、删除、重命名同时发生时损坏元数据。

输出文件按任务 ID 隔离。下载接口只允许访问输出目录和可预览资源目录，不能任意读取服务器路径。

## 配置与凭据

仓库只提交 `.env.example`、示例 TOML 或 README 配置说明。真实配置通过环境变量或用户本地 `.env` 提供，并默认被 `.gitignore` 排除。

必需配置包括：

1. 数据目录。
2. FFmpeg 路径或系统依赖检查。
3. 服务监听地址和端口。

可选配置包括：

1. Fish Speech 服务地址。
2. Edge TTS 默认参数。
3. LLM、ASR、在线素材服务 API key。
4. 任务队列后端。

可选外部服务不可用时，前端显示明确状态，后端返回结构化错误，核心本地混剪能力不因此整体不可用。

## 核心工作流

### 混剪任务

1. 用户选择本地素材或上传素材。
2. 用户填写脚本、字幕模板、音色、BGM、画幅、清晰度、输出参数。
3. 前端提交任务创建请求。
4. 后端保存任务快照，进入队列。
5. worker 准备素材、生成字幕、生成或引用配音、混合 BGM、调用渲染链路。
6. 前端轮询任务状态。
7. 任务成功后提供输出下载；任务失败时展示失败阶段和可操作原因。

任务创建时必须保存完整配置快照，避免用户后续修改模板或 BGM 后影响已提交任务。

### 字幕模板

1. 模板以结构化配置保存，支持导入导出。
2. 预览使用小样本字幕和目标画幅生成。
3. 正式导出以 ASS/libass 渲染为准。
4. DSL v2 可以作为配置层，旧模板能力通过兼容转换保留。

第一版不引入新的字幕场景图渲染器，避免同时维护多套真值。

### BGM 管理

1. 用户上传音频文件。
2. 后端解析时长、文件名、分类和显示名。
3. 前端支持试听、筛选、重命名、分类迁移和删除。
4. 混剪任务引用 BGM 的稳定 ID，而不是直接引用可变文件名。

删除分类时，分类下 BGM 迁移到未分类，不直接删除音频文件。

### 音色中心

1. Edge TTS 音色作为内置或可动态拉取的基础列表。
2. Fish Speech 音色通过可选外部服务适配。
3. 试听接口统一返回可播放资源或结构化错误。
4. 声音复刻入口保留，但当 Fish Speech 未配置时显示禁用状态和配置提示。

音色资源与混剪任务解耦。任务只保存音色 ID、参数和生成结果引用。

## 错误处理

后端统一返回结构化错误：

```json
{
  "code": "BGM_FILE_UNSUPPORTED",
  "message": "不支持的音频格式",
  "details": {
    "allowed": ["mp3", "wav", "m4a"]
  }
}
```

前端按错误类型展示中文提示：

1. 用户可修复错误：展示原因和下一步操作，例如文件格式不支持、外部服务未配置。
2. 系统依赖错误：展示缺失依赖，例如 FFmpeg 不可用。
3. 任务执行错误：展示失败阶段、日志摘要、可重试入口。
4. 未知错误：展示请求 ID 或任务 ID，方便定位日志。

失败任务保留配置快照和日志摘要，便于用户修改后重新提交。

## 前端设计原则

界面定位是工作台，不是营销页。第一版优先高密度、可扫描、可重复操作：

1. 导航稳定，不做复杂动效。
2. 按资源类型提供清晰列表和批量操作。
3. 所有关键操作有加载、成功、失败、空状态。
4. 试听、预览、下载使用熟悉图标按钮，并提供可访问名称。
5. 移动端保留核心创建任务能力，但复杂编辑可以采用分步表单和折叠区。
6. 页面文案使用中文，不出现旧系统的登录、退出、权限、个人网盘提示。

## 测试策略

第一版至少覆盖：

1. 配置加载测试：缺省配置、可选服务未配置、非法路径。
2. BGM 服务测试：上传、分类、重命名、删除分类迁移、试听元数据。
3. 字幕模板测试：模板保存、导入导出、ASS 输出关键字段。
4. 音色服务测试：Edge TTS 列表、Fish Speech 未配置降级、试听错误映射。
5. 混剪任务测试：任务创建快照、状态流转、失败状态、输出路径限制。
6. API 测试：主要路由的成功和错误响应。
7. 前端测试：React 组件渲染、核心中文文案、无旧登录态调用、移动端关键布局样式、Vite 构建产物。

需要真实 FFmpeg 或外部 Fish Speech 的测试标记为集成测试，默认本地单元测试不依赖这些外部服务。

## 实施阶段

### 阶段 1：产品骨架

建立仓库结构、Python 后端项目、React + Vite 前端入口、配置模板、数据目录约定、Docker/本地启动脚本、健康检查。

验收点：服务可启动，首页可访问，健康检查能报告 FFmpeg 和可选外部服务状态。

### 阶段 2：资源中心

实现 BGM 管理、音色中心、字幕模板库。优先完成这些可独立验证的模块。

验收点：用户能上传和试听 BGM，查看音色状态，保存和预览字幕模板。

### 阶段 3：混剪任务流

实现素材选择、任务创建、任务队列、渲染执行、状态轮询、输出下载。

验收点：用户能从素材和资源中心创建一个混剪任务，并下载渲染结果。

### 阶段 4：产品化打磨

补齐错误诊断、空状态、移动端适配、README、运行文档、测试覆盖和打包发布说明。

验收点：新用户能按 README 在本地启动，完成一次端到端混剪任务。

## 风险与约束

1. 源项目存在真实凭据和内网配置，新仓库抽取时必须只读参考，不能复制配置值。
2. 源项目的视频路由和服务文件较大，直接搬迁会继承耦合，新仓库必须按职责拆分。
3. Fish Speech、LLM、ASR、在线素材服务都属于可选外部依赖，不能成为本地核心链路的启动前提。
4. 视频渲染依赖 FFmpeg、字体、libass 和操作系统能力，Docker 与本地启动都要做依赖检查。
5. 移动端编辑复杂度较高，第一版移动端目标是可创建和查看任务，不追求完整桌面级精细编辑体验。

## 后续决策

已经确认的前端决策：

1. 首版采用 React + Vite + TypeScript，而不是手写纯 HTML/CSS/JS。
2. 构建产物仍按静态 SPA 部署，由 FastAPI 或 nginx 托管，不引入 Next.js SSR/Node 服务。
3. 后端第一版默认采用文件系统 + JSON 元数据，等权限管理和多人协作阶段再评估数据库。

这个选择保留组件化、状态管理、任务轮询和后续复杂编辑能力，同时避免 Next.js 带来的额外服务和部署复杂度。
